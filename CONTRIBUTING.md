# Contributing

Thank you for considering a contribution to the AI Resume & Job Match Agent. The project
accepts focused, tested changes that preserve explainability, candidate privacy, and Clean
Architecture boundaries.

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md). Report
vulnerabilities through the private process in [SECURITY.md](SECURITY.md), not through an issue
or pull request.

## Before you begin

- Search existing issues and pull requests before opening a new one.
- Open an issue for substantial API, scoring, persistence, or architecture changes so the design
  can be discussed before implementation.
- Never use a real resume, job application, API key, or other personal data in an issue, test,
  fixture, log, or commit.
- Keep the numeric match score deterministic. LLMs may extract and explain evidence, but they
  must not be the source of the score.

## Development setup

Prerequisites:

- Python 3.12 or 3.13
- [uv](https://docs.astral.sh/uv/)
- Docker with Docker Compose, for container and PostgreSQL workflows
- GNU Make is optional; every Make target maps to a documented `uv` command

Create a local environment:

```shell
git clone https://github.com/Miguesh/AI-Resume-Job-Match-Agent.git
cd AI-Resume-Job-Match-Agent
uv sync --all-groups --frozen
```

Copy `.env.example` to `.env` and keep the file untracked. The local AI provider works without
an external key. Only set `OPENAI_API_KEY` when deliberately testing the OpenAI adapter.

Initialize the database and run the API:

```shell
uv run alembic upgrade head
uv run uvicorn resume_matcher.main:app --reload
```

Install the optional commit hooks:

```shell
uvx pre-commit install
```

## Architecture rules

Dependencies point inward:

1. The domain layer owns entities, value objects, matching policy, and business invariants. It
   must not import FastAPI, SQLAlchemy, OpenAI, or file-format libraries.
2. The application layer coordinates use cases and declares ports. It depends on the domain,
   not on infrastructure implementations.
3. Infrastructure implements application ports for parsing, persistence, AI providers, storage,
   logging, and export.
4. The presentation layer translates HTTP requests and responses and obtains dependencies from
   the composition root. Business rules do not belong in routers.

Use separate domain entities, HTTP schemas, persistence models, and AI contracts. Validate data
at every boundary and pass dependencies explicitly where practical.

Architecturally significant choices should include an Architecture Decision Record. Call out any
new dependency, public API change, migration, retention behavior, prompt behavior, or security
tradeoff in the pull request.

## Quality checks

Run the same checks enforced by CI:

```shell
make ci
```

Without Make:

```shell
uv run ruff format --check src tests alembic
uv run ruff check src tests alembic
uv run mypy src
uv run pytest -m "not external" --cov=resume_matcher --cov-report=term-missing
```

Tests that require credentials or external services must carry the `external` marker and must
not run in the default suite. Prefer fakes and contract tests for AI integrations. Add or update:

- unit tests for domain rules and use cases;
- integration tests when adapters or component boundaries change;
- regression cases for scoring behavior;
- sanitized fixtures generated specifically for this project.

The project enforces branch coverage. A change that lowers coverage should normally add tests,
not weaken the threshold.

## Database migrations

Create an Alembic migration for every schema change:

```shell
uv run alembic revision --autogenerate -m "describe the schema change"
uv run alembic upgrade head
uv run alembic check
uv run alembic downgrade -1
uv run alembic upgrade head
```

Review autogenerated migrations manually. They must be deterministic, reversible when feasible,
and safe for existing data. Never edit an already released migration; add a new one.

## Pull requests

Create a focused branch and keep the pull request small enough to review. In the description:

- link the issue and state the user-visible outcome;
- explain architectural and security decisions;
- list the commands and scenarios used to verify the change;
- identify migrations, compatibility breaks, prompt changes, or operational risks;
- update API examples and documentation when behavior changes.

Do not commit generated exports, upload data, `.env` files, coverage output, or credentials. The
author is responsible for reviewing AI-assisted code and ensuring it is correct, licensed,
maintainable, and free of fabricated claims.
